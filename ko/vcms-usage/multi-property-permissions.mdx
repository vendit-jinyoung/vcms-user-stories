## 이런 상황 아닌가요?

> "2호점, 3호점 담당자가 다른데, 하나의 계정을 돌려 쓰고 있어요. 누가 뭘 바꿨는지 모르겠어요."

다지점 운영에서 계정·권한 관리는 보안과 운영 효율 모두에 직결돼요.

- **동시 접속하면 자꾸 튕겨요.** 여러 직원이 같은 계정으로 접속하면 세션 충돌이 생겨요.
- **퇴사 직원이 계정에 접근**해서 설정을 바꾼 적이 있어요. 비밀번호 변경만으로는 불안해요.
- **밤 근무 직원이 실수로 요금을 바꿨어요.** 요금 변경 권한은 관리자만 가져야 하는데, 계정이 하나라서 제한할 수가 없어요.
- **여러 직원이 동시에 재고를 바꿔서** 충돌이 생기고, "시스템이 오작동한다"고 오해했어요. 실제로는 사람 간 충돌이었어요.
- 여러 업장의 **ID가 꼬여 있어요.** 슬로우오션과 히든포레스트의 ID가 분리되어 있고 벤디트 직원 개인 ID와 얽혀있어요.
- **여러 사용자가 동시에 재고를 바꿔서 충돌이 생겼어요.** "시스템 오류"라고 생각했는데 실제로는 사람 간 충돌이었어요.
- **지점 ID가 꼬여 있어요.** 두 지점의 계정이 분리되어 있고 담당자 개인 ID와 얽혀서 어떤 계정으로 접속해야 하는지 모르겠어요.

## VCMS에서 이렇게 해결해요

<Check>
**사용자별 개별 계정**을 만들어서 동시 접속 시 충돌을 방지해요. 한 계정을 공유하지 않으니 "누가 바꿨는지" 추적이 가능해요.
</Check>

<Check>
**역할별 권한 설정**으로 프론트 직원에게는 예약 조회만, 관리자에게는 요금 변경까지 — 필요한 권한만 부여할 수 있어요.
</Check>

<Check>
마스터 계정에서 **전체 지점의 운영 현황을 한눈에** 파악할 수 있어요. 현장에 없어도 원격으로 관리 가능해요.
</Check>

<Warning>
여러 사용자가 **동시에 같은 객실의 재고를 변경하면** 데이터 충돌이 발생할 수 있어요. 재고 변경 권한은 지점 관리자 이상에게만 부여하세요.
</Warning>

## 이렇게 설정하세요

<Steps>
  <Step title="지점별 계정 구조 설계">
    운영 중인 지점과 필요한 사용자 수를 정리하세요.

    | 역할 | 접근 범위 | 주요 권한 |
    |------|----------|----------|
    | **사장/대표** | 전체 지점 | 모든 기능 + 사용자 관리 + 설정 |
    | **지점 관리자** | 해당 지점만 | 예약 + 재고 + 요금 + 보고서 |
    | **프론트 직원** | 해당 지점만 | 예약 조회 + 체크인/아웃 |
    | **청소/시설** | 해당 지점만 | 객실 상태 변경만 (해당 시) |
  </Step>

  <Step title="마스터 계정 설정">
    사장님(또는 총 관리자)의 마스터 계정을 설정하세요. 모든 지점의 데이터를 통합 조회하고, 사용자 관리 권한을 가져요.
  </Step>

  <Step title="지점별 관리자 계정 생성">
    각 지점의 관리자 계정을 만드세요. 해당 지점의 예약·재고·요금을 관리할 수 있지만, **다른 지점 데이터에는 접근 불가**하도록 제한해요.
  </Step>

  <Step title="직원 계정 생성 및 최소 권한 부여">
    프론트 직원 등 일반 사용자 계정을 만들고, **최소 권한 원칙**으로 필요한 권한만 부여하세요. 처음에는 조회 권한만 주고, 필요에 따라 추가하는 게 안전해요.
  </Step>

  <Step title="동시 접속 테스트">
    각자 다른 계정으로 동시 접속해서 충돌이 없는지 테스트하세요. 같은 화면을 동시에 편집하지 않도록 **업무 시간/구역을 분리**하는 규칙을 정하세요.
  </Step>

  <Step title="퇴사 직원 처리 프로세스 수립">
    직원이 퇴사하면 **당일 계정 비활성화**가 원칙이에요. 비밀번호 변경만으로는 부족해요 — 계정 자체를 비활성화해야 재접근을 막아요.
  </Step>
</Steps>

## 자주 틀리는 부분

<AccordionGroup>
  <Accordion title="직원 온보딩/오프보딩 체크리스트">
    **신규 직원 입사 시:**
    1. 직원 전용 계정 생성 (공용 계정 공유 금지)
    2. 역할에 맞는 최소 권한 부여
    3. 기본 조작 교육 (방막기, 예약 확인 위주)
    4. 비밀번호 정책 안내

    **직원 퇴사 시:**
    1. 당일 계정 비활성화
    2. 공용 비밀번호가 있었다면 즉시 변경
    3. 해당 직원이 마지막으로 변경한 설정 점검
    4. 분기마다 전체 계정 목록 감사
  </Accordion>

  <Accordion title="VCMS 탭을 여러 개 열면 세션이 공유돼요">
    같은 브라우저에서 VCMS 탭을 여러 개 열면 세션이 공유돼요. 다른 지점을 동시에 보고 싶다면 **크롬 시크릿 모드**나 **다른 브라우저**를 사용하세요. 또는 각 지점용 계정으로 따로 로그인하면 돼요.
  </Accordion>

  <Accordion title="재고 충돌이 자꾸 나는데 시스템 오류인가요?">
    대부분 **사람 간 충돌**이에요. 여러 직원이 동시에 같은 객실의 재고를 바꾸면 마지막에 저장한 값이 적용돼요. 재고 변경 권한은 한 사람(또는 소수)에게만 부여하고, 나머지는 조회만 가능하도록 제한하세요. "시스템이 맞다, 사람이 문제다"는 원칙을 기억하세요.
  </Accordion>

  <Accordion title="사장님이 현장에 없을 때 어떻게 모니터링하나요?">
    마스터 계정으로 VCMS에 접속하면 모든 지점의 예약 현황, 재고, 매출을 볼 수 있어요. 모바일 웹에서도 접속 가능하니까, 외출 중에도 확인할 수 있어요. 다만 모바일에서는 요금 변경 기능이 제한될 수 있어요.
  </Accordion>

  <Accordion title="지점별로 채널 수수료가 다른데요?">
    지점별로 채널 수수료율이 다를 수 있어요. 각 지점의 VCMS에서 채널 전송가를 개별 설정하세요. 마스터 계정에서 지점별 채널 수수료 현황을 비교 조회할 수도 있어요.
  </Accordion>

  <Accordion title="여러 사용자가 동시에 재고를 바꾸면 어떻게 되나요?">
    마지막에 저장한 값이 적용돼요. 두 사람이 동시에 같은 객실의 재고를 바꾸면, 나중에 저장한 사람의 값으로 덮어씌워져요. 이 때문에 "내가 바꿨는데 원래대로 돌아갔다"는 혼란이 생겨요. 재고 변경 권한은 한 사람에게만 부여하거나, 변경 시간대를 분리하는 규칙을 만드세요.
  </Accordion>

  <Accordion title="지점 ID가 꼬여서 어떤 계정으로 접속해야 할지 모르겠어요">
    지점별 계정 구조를 처음부터 명확하게 설계하지 않으면 ID가 꼬이는 경우가 생겨요. 현재 상황을 정리하려면:
    1. 현재 사용 중인 모든 계정 목록 작성
    2. 각 계정이 어느 지점에 속하는지 확인
    3. 불필요한 계정 비활성화
    4. 지점별 마스터 계정 재설정

    VCMS 고객지원에 계정 정리 지원을 요청할 수 있어요.
  </Accordion>
</AccordionGroup>

## 이 설정으로 해결된 사례

<Note>
**3지점 운영 모텔 L — "퇴사 직원 보안 사고 후 계정 체계를 완전히 바꿨어요"**

3개 지점을 운영하면서 모든 지점이 하나의 계정을 공유하고 있었어요. 퇴사한 직원이 이전 비밀번호로 접속해서 설정을 바꾼 적이 있었고, 그 후 VCMS에서 지점별·역할별 계정을 분리했어요.

사장님은 마스터 계정으로 3개 지점 전체를 보고, 각 지점 관리자는 자기 지점만 관리. 프론트 직원은 예약 조회와 체크인만 가능. "이제 누가 뭘 바꿨는지 바로 알 수 있어요. 특히 밤 근무자가 실수로 요금을 바꾸는 일이 완전히 사라졌어요."
</Note>
