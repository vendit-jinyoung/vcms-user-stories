# 떠나요 채널 운영 노트

떠나요는 주로 펜션 업장에서 사용되는 채널 관리 시스템으로, 자체 운영 기능과 OTA 판매 대행을 겸하는 독특한 구조를 가지고 있다. VCMS와의 연동 시 여러 고유한 제약과 주의점이 존재한다.

---

## 채널 특성

| 항목 | 내용 |
|------|------|
| 채널 유형 | 자체 운영 기능 + OTA 판매 대행 겸용 |
| 연동 방식 | 리피터 프로그램 기반 (PC 상시 가동 필요) |
| 인증 방식 | IP 인증 + 2차 인증 (기기 인증) |
| 요금 구조 | 타입 기반 (호실별 아님) |
| 재고 관리 | '열기' 또는 '막기'만 지원 — 세부적 수량 조절 불가 |
| 예약 처리 | 다박 예약을 1박 단위로 분할 처리 |
| 수수료 | 기본 플랫폼 수수료 + 판매 대행 수수료 (약 13%) |
| 요금 제한 | 채널 자체 요금 상한선 존재 |

---

## 많이 막히는 지점

### 1. 2차 인증 및 IP 인증

떠나요 연동의 가장 빈번한 장애 요인이다.

<AccordionGroup>
<Accordion title="IP 인증 문제">
- 여러 컴퓨터에서 접속 시 매번 IP 인증을 해야 하는 번거로움
- 9월부터 이 인증 방식이 더욱 강화된 적 있음
- 새로운 기기에서 접속하면 기존 인증이 해제될 수 있음
</Accordion>

<Accordion title="2차 인증 문제">
- 6월부터 인증 시스템이 도입되어 PC를 항상 켜두고 인증 번호를 입력해야 함
- 업장 PC에서만 기기 인증이 가능 — 원격에서 설정 불가
- 2차 인증을 설정하지 않았다고 주장하는 고객도 있으나, 기기 인증 문제로 확인되는 경우가 많음
</Accordion>

<Accordion title="리피터 프로그램">
- 떠나요 연동을 위해 리피터 프로그램이 PC에 상시 실행되어야 함
- 노트북을 닫으면 프로그램 연결이 끊김
- 오래된 컴퓨터에서는 성능 저하 및 접속 불가 문제 발생
- 리피터가 로그인되어 있지 않으면 연동에 문제가 발생
</Accordion>
</AccordionGroup>

<Warning>
떠나요 연동은 **PC 상시 가동**이 전제 조건이다. 시스템이 꺼져 있을 때 들어온 예약이 누락될 수 있으며, 노트북 사용 시 덮개를 닫으면 연결이 끊긴다. 전용 PC를 권장한다.
</Warning>

### 2. 요금 상한선

<Warning>
떠나요 플랫폼은 **요금 상한선**이 엄격하다. VCMS에서 설정한 높은 요금이 이 상한선을 초과할 경우, 해당 객실이 떠나요에서 자동으로 막히거나 최대가로만 적용된다. 유연한 요금 관리가 어렵다.
</Warning>

- VCMS에서 요금을 올려도 떠나요 상한선에 걸리면 반영되지 않음
- 무조건 할인 형태로만 요금이 설정되어 요금 인상 전략에 제약
- 성수기 요금 설정 시 특히 주의 필요

### 3. 타입 기반 요금 구조의 혼란

떠나요는 호실별이 아닌 **타입별**로 요금을 설정한다. 이로 인해:

| 문제 | 상세 |
|------|------|
| 동일 타입 내 요금 불일치 | VIP 원룸(402~405호)의 요금이 동일해야 하나 65,000원과 120,000원으로 다르게 표시되는 사례 |
| 요금 자동 변경 | 동일 타입 내에서 요금이 다르면 시스템이 계속 요금을 변경하려는 현상 발생 |
| 야놀자/떠나요 데이터 구조 차이 | 외부 플랫폼의 데이터 구조가 타입별로 분류되지 않아 요금 조작이 복잡 |

### 4. 수기 재입력 워크플로우

떠나요를 주 관리 시스템으로 사용하는 업장에서 빈번하게 발생하는 패턴이다.

<AccordionGroup>
<Accordion title="수기 재입력의 실태">
- 야놀자, 여기어때 등 다른 채널에서 들어온 예약을 떠나요에 수기로 다시 입력하는 운영 방식이 흔함
- 이는 오버부킹 및 취소 관리의 어려움을 야기
- VCMS 연동 시 수기 예약과 자동 연동 예약이 중복되어 데이터 정리가 필요
- 떠나요에서 취소한 예약이 벤디트 시스템에서도 삭제되어 예약이 유실되고 수동 재등록이 필요한 사례
</Accordion>
</AccordionGroup>

### 5. 다박 예약 처리 문제

<Warning>
떠나요는 다박 예약을 **1박 단위로 분할**하여 처리한다. 예를 들어 15박 예약이 시스템에서 15개의 개별 1박 예약으로 나뉘어 표시되며, 예약 박수 계산 방식도 VCMS와 달라 혼란을 야기한다. 수동 병합이 필요하다.
</Warning>

## 요금·재고 관점의 주의점

### 요금 설정 주의사항

| 항목 | 상세 |
|------|------|
| 상한선 확인 | 설정 전에 떠나요의 요금 상한선을 파악하고, 상한선 이내에서 요금 설정 |
| 수수료 제외 입력 | 떠나요에 금액 입력 시 수수료를 제외하고 입력해야 함 — 자동 처리되지 않음 |
| 타입별 통일 | 동일 타입 내 모든 호실의 요금을 통일해야 시스템 오류 방지 |
| 시즌 요금 설정 | 시즌 요금 설정이 복잡하여 벤디트의 추가 지원이 필요한 경우 많음 |

### 재고 관련

- 떠나요는 '열기' 또는 '막기'만 지원 — 특정 수량만 오픈하는 세부 조절 불가
- 야놀자 펜션과 떠나요는 객실 타입별로 재고를 관리하며, 여러 타입을 하나로 묶는 방식 때문에 오버부킹 위험이 높음
- VCMS에서는 떠나요 상품 기준으로 재고를 관리하며, 판매 시 자동으로 재고가 줄어드는 로직 적용

### 야놀자와 동시 판매 리스크

<Warning>
야놀자와 떠나요 채널이 서로 다른 사업자 등록으로 동일한 객실에 대해 경쟁적으로 예약이 들어오는 경우가 보고되었다. 수수료 이중 부과 가능성과 오버부킹 위험이 있으므로, 양 채널의 재고 연동 상태를 면밀히 확인해야 한다.
</Warning>

| 리스크 | 상세 |
|--------|------|
| 재고 충돌 | 야놀자와 떠나요에서 동시에 예약이 들어올 경우 오버부킹 |
| 호실 고정 | 떠나요 예약은 호실이 고정되어 VCMS에서 변경 불가 — 다른 채널의 방 배정에 제약 |
| 수수료 이중 부과 | 같은 객실에 대해 양 채널에서 수수료가 부과될 수 있음 |

## 먼저 확인할 순서

<Steps>
  <Step title="리피터 프로그램 실행 상태 확인">
    떠나요 리피터 프로그램이 업장 PC에서 정상적으로 실행 중인지 확인한다. 로그인 상태와 연결 상태를 점검.
  </Step>
  <Step title="PC 상시 가동 여부 확인">
    전용 PC가 있는지, 노트북을 닫지 않는지 확인. 시스템이 꺼지면 예약이 누락될 수 있다.
  </Step>
  <Step title="IP/기기 인증 상태 확인">
    현재 사용 중인 PC에서 IP 인증 및 기기 인증이 완료되어 있는지 확인. 다른 기기에서 접속했다면 재인증 필요.
  </Step>
  <Step title="요금 상한선 확인">
    떠나요의 요금 상한선이 현재 설정하려는 요금과 충돌하지 않는지 확인.
  </Step>
  <Step title="수기 예약 중복 정리">
    VCMS 연동 전에 떠나요에 수기로 입력한 다른 채널 예약이 있다면 중복 제거.
  </Step>
  <Step title="야놀자 재고 연동 상태 확인">
    야놀자와 떠나요가 동시에 같은 객실을 판매하고 있지 않은지 확인.
  </Step>
</Steps>

## 자주 오해하는 포인트

<AccordionGroup>
<Accordion title="떠나요에서 요금을 올리면 바로 반영된다?">
**상한선에 걸리면 반영되지 않는다.** VCMS에서 설정한 높은 요금이 떠나요의 상한선을 초과하면 해당 객실이 자동으로 막히거나 최대가로만 적용된다.
</Accordion>

<Accordion title="떠나요에서 직접 방을 막으면 VCMS에도 반영된다?">
**반영되지 않는다.** 떠나요 자체 사이트에서 직접 방을 막는 것은 VCMS와 연동되지 않는다. VCMS를 통해 관리해야 한다.
</Accordion>

<Accordion title="떠나요와 VCMS를 동시에 사용할 수 있다?">
**주의가 필요하다.** 떠나요는 자체 운영 기능이 강해 VCMS와 운영 기준이 충돌할 수 있다. 두 시스템을 동시에 사용하면 예약 관리 방식이 달라 혼선이 발생할 수 있으므로, 운영 기준을 먼저 정리하는 것이 좋다.
</Accordion>

<Accordion title="떠나요 채널에서 취소하면 VCMS에서도 자동 취소된다?">
**자동 취소가 되지만, 이것이 문제가 될 수 있다.** 떠나요 채널에서 취소한 예약이 다른 운영 시스템에서도 예상치 못하게 삭제되어 예약이 유실되는 사례가 보고되었다. 수동 재등록이 필요할 수 있다.
</Accordion>

<Accordion title="연박 예약이 하나의 예약으로 관리된다?">
**아니다.** 떠나요에서 들어온 연박 예약은 VCMS에서 1박 단위로 쪼개져 표시된다. 수동으로 병합해야 하며, 각 예약의 금액 및 날짜 변경 사항도 개별 대응이 필요하다.
</Accordion>
</AccordionGroup>

## 운영 팁

- **떠나요 예약은 호실 고정**: 떠나요 예약은 들어온 호실 그대로 배정하고, 여기어때나 야놀자 같은 다른 채널 예약은 떠나요 예약이 차지한 호실을 제외한 남는 방에 배정하는 것이 원칙이다.
- **떠나요 전용 마감 기능 활용**: VCMS에서 떠나요 상품만 따로 마감하는 기능이 있으므로, 다른 채널에 영향을 주지 않고 떠나요만 관리할 수 있다.
- **수수료 구조 인지**: 떠나요를 통한 판매는 기본 플랫폼 수수료에 판매 대행 수수료가 추가되어 전체 비용이 높아진다. 네이버의 자체 수수료(4%)와 비교하면 차이가 크다.
- **낮은 예약 점유율 확인**: 떠나요 채널의 예약 점유율이 5% 미만인 업장이 있으며, 예약 유입이 거의 없어 사용을 중단한 사례도 보고됨. 채널별 ROI를 정기 점검할 것.
- **캘린더 뷰 참고**: 떠나요의 캘린더 뷰는 직관적이고 색상으로 객실 상태를 구분할 수 있다는 긍정적 피드백이 있다. VCMS 사용 시에도 유사한 방식의 확인이 필요하다.
- **떠나요 사용 중단 시 리뷰 주의**: 떠나요와의 계약 해지 시 기존에 쌓아둔 리뷰가 모두 사라질 수 있다. 특히 여기어때와 떠나요가 분리되면서 리뷰 손실 피해 사례가 보고됨.
- **운영 기준 통일 검토**: 떠나요와 VCMS가 충돌하는 경우가 반복된다면, 어느 시스템을 기준으로 재고와 요금을 관리할지 내부 기준을 먼저 통일하는 것이 좋다.
